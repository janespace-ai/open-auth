# Open-Auth

**一个通用的 AI Agent 授权协议 —— 手机审批，Agent 执行。**

[English](../README.md)

```
┌──────────┐                                    ┌──────────────┐
│          │  "我需要删除                          │              │
│    AI    │   生产库里 15000 条                    │    你的手机   │
│   Agent  │   过期数据"                           │              │
│          │ ───────────────────────────────────▶ │  ┌────────┐  │
│          │           端到端加密                   │  │ 批准？  │  │
│          │                                      │  │        │  │
│          │ ◀─────────────────────────────────── │  │  ✓  ✗  │  │
│          │          批准 / 拒绝                   │  └────────┘  │
└──────────┘                                    └──────────────┘
```

AI Agent 正在替你做越来越多的事 —— 但你怎么保持控制权？Open-Auth 让任何 Agent 在行动前先征求你的许可。你在手机上审批（或拒绝），就像用 Google Authenticator 确认登录一样。

## 为什么需要 Open-Auth？

如今，每个 Agent 到人类的授权系统都是私有的。没有一个标准方式让 Agent 说"我需要你的批准"，然后你在可信设备上响应。

Open-Auth 的方案：

- **通用协议** —— 任何 Agent、任何授权端、任何操作
- **端到端加密** —— 中继服务器看不到任何内容，只有你和你的 Agent 共享密钥
- **基于能力模型** —— 授权端声明自己能做什么（签名交易、审批操作等），Agent 请求特定的操作
- **一对一配对** —— 用短码或扫 QR 一次配对，像蓝牙一样

## 应用场景

### Agent 要执行危险操作

你的运维 Agent 需要清理生产数据库的过期数据。

> **手机通知：** "维护 Agent 想要删除 `user_sessions` 表的 15,420 条记录。批准？"
>
> 你确认详情，点 **批准** → Agent 继续执行。

### Agent 需要 API 访问授权

你的调研 Agent 想调用付费 API。

> **手机通知：** "调研 Agent 请求访问 OpenAI API，预计花费 $5.00。批准？"
>
> 看起来合理，点 **批准** → Agent 获得一次性授权令牌。

### Agent 要签署加密货币交易

你的交易 Agent 发现了一个套利机会，想在 Uniswap 上执行交换。

> **手机通知：** "交易 Agent 想在 Base 链上用 1 ETH 兑换 USDC，预估价值 $3,200。批准？"
>
> 你核对详情，点 **批准** → Agent 拿到签名后的交易。

### Agent 请求任何你定义的权限

Open-Auth 不限于预定义的操作。任何能力都可以注册：

- 文件系统访问审批
- 邮件发送授权
- 基础设施变更（扩容、部署）
- Crypto 之外的金融操作

## 工作原理

```
     一次性配对                               每次请求
  ─────────────────────                ─────────────────────────────

  Agent              手机              Agent              手机
    │                  │                 │                  │
    │   输入短码        │                 │  "authorize"     │
    │   或扫 QR        │                 │  请求 (E2EE)     │
    │ ◄──────────────► │                 │ ───────────────► │
    │                  │                 │                  │
    │   密钥交换        │                 │                  │  用户
    │   (X25519)       │                 │                  │  审查
    │ ◄──────────────► │                 │                  │  并决策
    │                  │                 │                  │
    │   配对完成 ✓      │                 │  结果 (E2EE)     │
    │                  │                 │ ◄─────────────── │
    │                  │                 │                  │
```

1. **一次配对** —— Agent 和你的手机通过短码或 QR 交换密钥，建立加密通道。
2. **Agent 发起请求** —— 当 Agent 需要授权时，通过中继服务器发送加密请求。
3. **你来决定** —— 手机展示 Agent 想做什么，你批准、拒绝或忽略。
4. **Agent 获得结果** —— 如果批准，Agent 收到结果（签名、令牌、或简单的"是"）。

中继服务器是透明管道 —— 它只转发加密消息，看不到任何内容。

## 协议架构

Open-Auth 是一个 4 层协议栈：

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 3: 应用层                                             │
│  • capabilities 查询 —— "你能做什么？"                        │
│  • authorize 请求 —— "请做 X"                                │
│  • 三态响应 —— 批准 / 拒绝 / 错误                             │
├─────────────────────────────────────────────────────────────┤
│  Layer 2: 安全层                                             │
│  • 端到端加密 (X25519 + AES-256-GCM)                         │
│  • 防重放保护                                                │
│  • 三级重连验证                                               │
├─────────────────────────────────────────────────────────────┤
│  Layer 1: 传输层                                             │
│  • 三种消息类型: pair_complete / handshake / encrypted        │
│  • 传输无关 (HTTP 中继、WebSocket、蓝牙...)                   │
├─────────────────────────────────────────────────────────────┤
│  Layer 0: 配对层                                             │
│  • 短码或 QR 码配对                                          │
│  • X25519 密钥交换                                           │
│  • 确定性通道 ID 派生                                         │
└─────────────────────────────────────────────────────────────┘
```

完整规范详见：[`protocol/`](protocol/)

## 能力模型

Open-Auth 不硬编码 Agent 能请求什么。授权端声明**能力** —— 它支持的一组命名操作。

```
授权请求:
{
  "method": "authorize",
  "capability": "evm-signer",           ← 哪个能力
  "action": "sign_transaction",          ← 哪个操作
  "params": { "to": "0x...", ... },      ← 操作参数
  "context": {                           ← 帮助人类决策
    "description": "用 1 ETH 兑换 USDC",
    "estimatedUSD": 3200
  }
}
```

**内置参考能力：**

| 能力 | 操作 | 说明 |
|------|------|------|
| `evm-signer` | `sign_transaction`, `sign_message`, `sign_typed_data` | EVM 区块链签名 |
| `generic-approval` | `approve` | 任何 Agent 操作的人工审批 |

任何人都可以为自己的场景定义新能力。

## 兼容性

Open-Auth 从 [claw-wallet](https://github.com/anthropic/claw-wallet) 项目中抽象而来，完全向后兼容：

- **claw-wallet Desktop** 无需任何改动 —— 继续使用现有协议
- **claw-wallet Agent SDK** 可以在保持旧协议连接 Desktop 的同时，对新的授权端使用 Open-Auth
- **Go 中继服务器** 无需任何改动 —— 它只负责转发不透明消息

完整兼容指南：[`protocol/compatibility/claw-wallet.md`](protocol/compatibility/claw-wallet.md)
